### Component Responsibilities

| Component | Responsibility | Invocation |
|-----------|----------------|------------|
| `car_search.md` | Single source of truth for search criteria (budget, must-haves, nice-to-haves, context) | User-maintained, read by skills |
| `ev-search` skill | Reads `car_search.md`, queries EV sites, returns list of matching models | Manual: `/ev-search` |
| `ev-detail` skill | Fetches deep specs, reviews, and ownership signals for one named car | Manual: `/ev-detail volvo ex30` |
| `ev-compare` skill | Reads all `research/*.md` files and generates side-by-side comparison table | Manual: `/ev-compare` |
| `ev-sources` skill | Background knowledge: site URLs, fetch patterns, known data quirks per source | Auto-invoked by Claude when other EV skills run |

## Recommended Project Structure

```
car-research/
├── car_search.md                # Search criteria (user edits this)
├── comparison.md                # Generated by /ev-compare
├── research/
│   ├── volvo-ex30.md            # Generated by /ev-detail
│   ├── skoda-enyaq.md
│   └── [make-model].md
└── .claude/
    └── skills/
        ├── ev-search/
        │   ├── SKILL.md         # Search skill instructions
        │   └── output-template.md  # Template for result list format
        ├── ev-detail/
        │   ├── SKILL.md         # Detail fetch instructions
        │   └── car-template.md  # Template for per-car markdown output
        ├── ev-compare/
        │   ├── SKILL.md         # Comparison skill instructions
        │   └── comparison-template.md  # Template for table structure
        └── ev-sources/
            ├── SKILL.md         # Source site knowledge (auto-invoked)
            └── sites.md         # Per-site fetch patterns and quirks
```

### Structure Rationale

- `.claude/skills/<name>/SKILL.md`: Required by Claude Code skills standard. Directory structure allows bundling supporting files (templates) alongside each skill without cluttering `SKILL.md` itself.
- `car_search.md` at repo root: User-facing, not buried. Skills read it via relative path. Editing it changes behaviour of all skills without touching skill files.
- `research/` as dedicated output directory: Keeps generated files separate from user-maintained files. Makes `/ev-compare` glob pattern trivial (`research/*.md`). Easy to version-control or gitignore individual files.
- `output-template.md` / `car-template.md` per skill: Keeps `SKILL.md` under 500 lines (Claude Code recommendation). Claude references the template file when generating output.
- `ev-sources/` as separate skill with `user-invocable: false`: Site-specific fetch knowledge (base URLs, known URL patterns, Danish-market quirks) doesn't need to be duplicated across skills. Claude loads it automatically when EV skills are active.

## Architectural Patterns

### Pattern 1: Input File as Configuration

**What:** A user-maintained markdown file (`car_search.md`) acts as the sole configuration surface. Skills read it at invocation time using dynamic context injection.

**When to use:** When you want the skill to be generic (not hardcoded for one person's criteria) but the user's needs to evolve without editing skill files.

**Trade-offs:** Simple for the user; skills stay reusable. Requires skills to parse freeform markdown reliably, which Claude handles well but is not strict validation.

**Example (in SKILL.md):**
```yaml
---
name: ev-search
description: Search for EVs matching criteria in car_search.md
disable-model-invocation: true
allowed-tools: WebFetch, WebSearch, Bash, Read
---

Read car_search.md first:
!`cat car_search.md`

Using the criteria above, search the following sites for matching EVs...
```

### Pattern 2: Template Files for Consistent Output

**What:** Each skill that produces output references a `*-template.md` file in its skill directory. Claude fills in the template rather than inventing structure each time.

**When to use:** Any time output format needs to be consistent across multiple invocations (multiple cars must have the same sections so comparison works).

**Trade-offs:** Consistent, diffable output. Slightly more files to maintain. Worth it because `/ev-compare` depends on predictable section names in per-car files.

**Example (car-template.md excerpt):**
```markdown
# [Make Model] ([Year])

## Summary
[2-3 sentence overview]

## Key Specs
| Spec | Value |
|------|-------|
| Range (WLTP) | |
| Charging (AC) | |
| Turning radius | |
| Width | |
| Tow capacity | |

## Ownership Signals
- **Reliability:**
- **Parts cost (DK):**
- **Insurance notes:**

## Danish Market
- **Registration tax:**
- **Green owner tax:**
- **DK price range:**

## Sources
```

### Pattern 3: Background Knowledge Skill (Auto-Invoked)

**What:** An `ev-sources` skill with `user-invocable: false` holds site-specific knowledge — base URLs, known URL patterns, Danish-market nuances. Claude loads it automatically when the other EV skills are active.

**When to use:** Knowledge that every EV skill needs but that would bloat each individual skill if copied in. Sources change independently of skill logic.

**Trade-offs:** Clean separation of concerns. Slightly less obvious to someone reading a single skill in isolation. Worth it to avoid copy-paste drift across skills.

**Example frontmatter:**
```yaml
---
name: ev-sources
description: EV data source knowledge for ev-database.org, FDM, and greengarage.dk. Loaded automatically when researching EVs.
user-invocable: false
---
```

### Pattern 4: Manual-Only Skills with `disable-model-invocation`

**What:** All action skills (`ev-search`, `ev-detail`, `ev-compare`) have `disable-model-invocation: true` so Claude never decides to run a search or write a file unprompted.

**When to use:** Any skill that writes files, makes web requests, or has side effects the user controls timing of.

**Trade-offs:** User must explicitly invoke. Prevents unwanted file writes or spurious fetches during unrelated conversations in this repo.

## Data Flow

### Primary Flow: Search → Detail → Compare

```
User edits car_search.md
    ↓
/ev-search
    ↓ reads car_search.md via !`cat car_search.md`
    ↓ fetches ev-database.org, FDM, greengarage.dk
    ↓ returns list of candidate model names
    ↓
User reviews list, picks candidates
    ↓
/ev-detail volvo ex30
    ↓ fetches specs from ev-database.org/car/volvo/ex30
    ↓ fetches FDM test/review if available
    ↓ fetches greengarage.dk if relevant
    ↓ checks car_search.md criteria (turning radius, tow hitch, etc.)
    ↓ fills car-template.md
    ↓ writes research/volvo-ex30.md
    ↓
(repeat /ev-detail for each candidate)
    ↓
/ev-compare
    ↓ globs research/*.md
    ↓ extracts Key Specs and Ownership Signals sections
    ↓ generates comparison.md table
    ↓ writes comparison.md to repo root
```

### File Dependency Graph

```
car_search.md ──────→ ev-search (reads)
car_search.md ──────→ ev-detail (reads for criteria validation)
ev-detail ──────────→ research/[make-model].md (writes)
research/*.md ───────→ ev-compare (reads)
ev-compare ─────────→ comparison.md (writes)
ev-sources.sites.md → ev-search, ev-detail (background context)
car-template.md ────→ ev-detail (fills and writes)
```

### Key Data Flows

1. **Criteria propagation:** `car_search.md` is injected into both `ev-search` (to filter candidates) and `ev-detail` (to flag which must-haves a car satisfies). Skills always re-read the file at invocation so changing criteria takes effect immediately without touching skill files.

2. **Per-car file as the durable artifact:** `research/[make-model].md` is the canonical record for a car. It is written once by `ev-detail`, can be manually edited, and is read by `ev-compare`. The file is the handoff between skills — no shared in-memory state, no database.

3. **Comparison as derivation:** `ev-compare` is purely a reader. It derives its output from whatever `research/*.md` files exist. Adding a new car is just running `/ev-detail new-car-name`; the comparison skill picks it up automatically.

## Build Order (Skill Development Sequence)

Skills have dependencies that determine build order:

```
Phase 1: Foundation
  ev-sources/ ← No dependencies. Defines site knowledge all others use.
  car-template.md ← No code, just a template. Defines output contract.

Phase 2: Detail fetch (core value)
  ev-detail/ ← Depends on: ev-sources (site patterns), car-template.md (output format)
               This is the highest-value skill. Build and validate first.

Phase 3: Comparison (depends on detail output existing)
  ev-compare/ ← Depends on: research/*.md files (produced by ev-detail)
                Cannot be usefully tested until ev-detail has run at least once.

Phase 4: Search (discovery, depends on knowing what good output looks like)
  ev-search/ ← Depends on: ev-sources (site patterns)
               Can be built independently, but lower priority than detail+compare.
               User can manually identify candidate models to bootstrap.
```

**Rationale for this order:** The detail skill (`ev-detail`) and its output template (`car-template.md`) define the data contract the entire system depends on. Building it first validates that the sites provide fetchable data and that the output format is useful. `ev-compare` cannot be tested meaningfully without real per-car files. `ev-search` is a convenience that helps discover candidates but is not required — the user can supply model names directly.

## Anti-Patterns

### Anti-Pattern 1: Hardcoded Criteria in Skill Files

**What people do:** Embed budget range, must-have features, or Danish tax assumptions directly in `SKILL.md`.

**Why it's wrong:** Skills become personal rather than generic. Criteria change (budget shifts, a must-have becomes a nice-to-have). User must edit skill files instead of `car_search.md`. Skill files get out of sync with actual needs.

**Do this instead:** Always read `car_search.md` at invocation time via `!`cat car_search.md``. Treat skills as behaviour, treat `car_search.md` as configuration.

### Anti-Pattern 2: Monolithic SKILL.md with All Logic

**What people do:** Put site-specific URL patterns, output templates, and fetch logic all inside a single large `SKILL.md`.

**Why it's wrong:** `SKILL.md` bloats past 500 lines (Claude Code's recommended ceiling). Context window fills with source knowledge even when not needed. Hard to update one source without touching everything.

**Do this instead:** Separate concerns into supporting files. `ev-sources/sites.md` for source knowledge. `car-template.md` for output structure. Keep `SKILL.md` as an orchestration script that references these files.

### Anti-Pattern 3: Auto-Invocable Action Skills

**What people do:** Leave `disable-model-invocation` unset on skills that write files or make web requests.

**Why it's wrong:** Claude may decide to run `/ev-detail` or `/ev-compare` mid-conversation when you ask something tangentially related. Unexpected file writes, stale comparisons, or fetches that consume context budget.

**Do this instead:** Set `disable-model-invocation: true` on all skills with side effects. Only `ev-sources` (pure knowledge, no writes) should be auto-invocable.

### Anti-Pattern 4: Inconsistent Per-Car File Structure

**What people do:** Let each invocation of `ev-detail` generate whatever sections Claude decides are appropriate for that car.

**Why it's wrong:** `ev-compare` depends on consistent section names and table structure to extract data. If one car file has "Reliability" and another has "Quality", the comparison breaks or requires manual cleanup.

**Do this instead:** Use `car-template.md` as a strict template. Instruct `ev-detail` to always fill the template, not invent structure. Sections that have no data should be explicit ("not available" or "not found") rather than omitted.

## Integration Points

### External Services

| Service | Integration Pattern | Notes |
|---------|---------------------|-------|
| ev-database.org | WebFetch on known URL patterns (e.g. `/car/make/model`) | High-confidence specs data. URL structure is stable. Danish market prices available. |
| fdm.dk/tests, fdm.dk/nyheder | WebSearch for model name + site filter, then WebFetch on result URLs | FDM does not have predictable per-car URLs. Search first, then fetch review page. |
| greengarage.dk | WebSearch for model name + site filter, then WebFetch | Similar to FDM — search-then-fetch pattern. |
| Danish tax registries | WebSearch for current registration tax for specific model | Not a stable API. Best-effort via search. Flag as "verify before purchase". |

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| `car_search.md` → skills | Shell injection via `!`cat car_search.md`` | File read at invocation time. No caching. Changes take effect immediately. |
| `ev-detail` → `research/*.md` | Claude writes file using Write tool | Per-car file is the durable handoff. Can be manually edited after generation. |
| `research/*.md` → `ev-compare` | Claude reads files using Read + Glob | Glob pattern `research/*.md` picks up all cars. Order is filesystem-dependent; comparison skill should sort by model name. |
| `ev-sources` → other skills | Automatic context loading (auto-invoked skill) | Claude Code loads `ev-sources` description into context when other EV skills are active. Full content loads on invocation. |
| `car-template.md` → `ev-detail` | Skill references template file in instructions | Claude reads template, fills it, writes output. Template is the schema contract. |

## Scaling Considerations

This is a local research tool, not a service. Scaling concerns are different:

| Concern | Now (5-10 cars) | Later (30+ cars) |
|---------|-----------------|-------------------|
| Context size | Each per-car file is small; comparison reads all of them | If `research/*.md` grows very large, `ev-compare` may need to be selective (e.g. compare a named subset) |
| Fetch rate | Sequential fetches per `/ev-detail` invocation | No parallelization needed at this scale; Claude's sequential fetching is fine |
| Output staleness | Re-run `/ev-detail make model` to refresh a file | Add a "last updated" field to `car-template.md` so comparison table shows data freshness |
| New sources | Add site patterns to `ev-sources/sites.md` | No structural change required; skill files stay the same |
